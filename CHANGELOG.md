## Version Numbering

**Format:** MAJOR.MINOR.PATCH

- **MAJOR**: Breaking changes
- **MINOR**: New features (backward compatible)
- **PATCH**: Bug fixes (backward compatible)

**Examples:**
- 0.1.0 - Initial development release
- 0.2.0 - Added combat system
- 0.2.1 - Fixed combat bugs
- 1.0.0 - First stable release
- 2.0.0 - Major rewrite with breaking changes

---

## Release Schedule

**Development Releases (0.x.x):**
- Frequent updates
- Breaking changes allowed
- Beta/Alpha quality

**Stable Releases (1.x.x+):**
- Quarterly minor releases
- Monthly patch releases
- No breaking changes in minor/patch

**Long-Term Support:**
- Major versions supported for 1 year
- Critical security fixes only

## Changelog Maintenance

### When to Update
- Every commit that changes functionality
- Before creating a release
- When closing issues/PRs

### How to Update
1. Add entry under [Unreleased]
2. Use appropriate category (Added/Changed/Fixed/etc.)
3. Link to issue/PR numbers
4. Use clear, user-focused language
5. When releasing, move to version section

### Entry Format
```markdown
- Brief description of change (#123)
- Another change that fixes a bug (#124)
- New feature added to X system (#125)
```
